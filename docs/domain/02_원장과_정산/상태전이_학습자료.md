# 상태전이 학습자료

관련 Jira: [SPN-33](https://aslan0.atlassian.net/browse/SPN-33)

이 문서는 StablePay에서 자주 등장하는 `상태`, `전이`, `계정 타입`, `정산`, `출금` 흐름을 한 번에 구분하기 위한 학습자료다.

## 이 문서를 만든 이유

현재 프로젝트에서는 다음과 같은 값들이 함께 등장한다.

| 구분 | 예시 |
| --- | --- |
| Payment 상태 | `PENDING`, `ONCHAIN_DETECTED`, `FINALIZED`, `FAILED` |
| Ledger 계정 타입 | `CUSTOMER`, `MERCHANT_PENDING`, `PLATFORM_FEE` |
| Settlement 상태 | `DRAFT`, `READY`, `APPROVED`, `PAID`, `FAILED` |
| Withdrawal 상태 | `REQUESTED`, `APPROVED`, `SIGNED`, `BROADCASTED`, `CONFIRMED`, `FAILED` |

겉으로 보면 모두 상태처럼 보이지만, 실제 의미는 서로 다르다.

가장 중요한 구분은 아래와 같다.

| 구분 | 무엇을 나타내는가? | 쉬운 비유 |
| --- | --- | --- |
| 상태 | 하나의 업무가 어디까지 진행되었는가 | 택배 배송 상태 |
| 계정 타입 | 돈을 어느 성격의 장부 칸에 기록하는가 | 회계 장부의 계정 과목 |
| 상태 전이 | 한 상태에서 다음 상태로 넘어가는 규칙 | 배송 준비 -> 배송 중 -> 배송 완료 |

즉, `PENDING`은 결제가 어디까지 진행되었는지를 말하지만, `MERCHANT_PENDING`은 가맹점에게 지급 예정인 돈을 기록하는 원장 계정의 역할을 말한다.

## 전체 그림

```plaintext
사용자 결제
   |
   v
Payment 상태
PENDING -> ONCHAIN_DETECTED -> FINALIZED
                                   |
                                   v
                              Ledger 기록
                 CUSTOMER / MERCHANT_PENDING / PLATFORM_FEE
                                   |
                                   v
                              Settlement 상태
                     DRAFT -> READY -> APPROVED -> PAID
                                   |
                                   v
                              Withdrawal 상태
             REQUESTED -> APPROVED -> SIGNED -> BROADCASTED -> CONFIRMED
```

이 흐름에서 주의해야 할 점은 Payment, Ledger, Settlement, Withdrawal이 하나의 상태 머신으로 합쳐지는 것이 아니라는 점이다.

각 도메인은 자기만의 상태 또는 타입을 가진다. 다만 앞 단계의 결과가 다음 단계의 입력이 된다.

## 1. Payment 상태

Payment는 결제 자체의 진행 상태를 나타낸다.

예를 들어 사용자가 10 USDC를 결제했다면, Payment는 "이 결제가 생성되었는가?", "온체인에서 감지되었는가?", "충분히 확정되었는가?"를 추적한다.

| 상태 | 의미 | 다음으로 갈 수 있는 상태 |
| --- | --- | --- |
| `PENDING` | 결제 요청은 만들어졌지만 아직 온체인 transaction을 감지하지 못한 상태 | `ONCHAIN_DETECTED`, `FAILED` |
| `ONCHAIN_DETECTED` | transaction hash 또는 event를 감지했지만 아직 finality가 충분하지 않은 상태 | `FINALIZED`, `FAILED` |
| `FINALIZED` | confirmation/finality 기준을 만족해서 결제를 확정할 수 있는 상태 | Ledger 기록 생성 대상 |
| `FAILED` | 잘못된 transaction, 금액 불일치, 토큰 불일치 등으로 실패한 상태 | 일반적으로 종료 상태 |

Payment 상태 전이는 아래처럼 이해하면 된다.

```plaintext
PENDING
  |
  | 온체인 tx/event 감지
  v
ONCHAIN_DETECTED
  |
  | confirmation/finality 확보
  v
FINALIZED

어느 단계에서든 검증 실패 시 FAILED
```

현재 Phase 1에서는 사람이 API로 Payment 상태를 바꾼다. Phase 2에서는 Event Indexer가 블록체인을 읽고 이 상태 변경을 자동화하는 방향으로 확장한다.

## 2. Ledger 계정 타입

Ledger는 Payment처럼 "진행 상태"를 나타내는 곳이 아니다.

Ledger는 돈이 어느 계정에서 어느 계정으로 이동했는지를 기록하는 장부다.

이때 `AccountType`은 그 계정이 어떤 역할의 돈을 담는지를 구분한다.

| 계정 타입 | 의미 | 예시 |
| --- | --- | --- |
| `CUSTOMER` | 고객이 지불한 돈이 빠져나가는 계정 역할 | 고객 결제 금액 10 USDC 차감 |
| `MERCHANT_PENDING` | 가맹점에게 귀속되었지만 아직 정산/지급되지 않은 금액 | 가맹점 지급 예정 9.8 USDC |
| `PLATFORM_FEE` | 플랫폼 수수료로 귀속되는 금액 | StablePay 수수료 0.2 USDC |

중요한 점은 `MERCHANT_PENDING`이 Payment 상태의 `PENDING`과 다르다는 것이다.

| 이름 | 속한 도메인 | 의미 |
| --- | --- | --- |
| `Payment.PENDING` | Payment | 결제가 아직 온체인에서 감지되지 않음 |
| `AccountType.MERCHANT_PENDING` | Ledger | 가맹점에게 지급 예정인 돈을 기록하는 계정 |

## 3. Ledger에서 row는 언제 생기는가?

10 USDC 결제가 확정되고, 플랫폼 수수료가 0.2 USDC라고 해보자.

이때 보통 새로 생기는 것은 `ledger_transactions` 1 row와 `ledger_entries` 여러 row다.

`ledger_accounts`는 결제마다 매번 새로 만드는 row가 아니다. 보통 `owner_id + account_type + currency` 기준으로 미리 있거나, 필요할 때 한 번 만들어 재사용한다.

```plaintext
ledger_accounts
  - customer_1 / CUSTOMER / USDC
  - merchant_1 / MERCHANT_PENDING / USDC
  - stablepay / PLATFORM_FEE / USDC

payment pay_123 FINALIZED
  |
  v
ledger_transactions
  - id: ltx_123
  - reference_type: PAYMENT
  - reference_id: pay_123

ledger_entries
  - customer_1 / DEBIT  / 10.0 USDC
  - merchant_1 / CREDIT / 9.8 USDC
  - stablepay  / CREDIT / 0.2 USDC
```

정리하면 아래와 같다.

| 테이블 | 결제 1건마다 새로 생기는가? | 이유 |
| --- | --- | --- |
| `ledger_accounts` | 보통 아니다 | 계정은 돈을 담는 장부 칸이라 여러 거래에서 재사용한다 |
| `ledger_transactions` | 보통 1개 생긴다 | 하나의 비즈니스 사건을 묶는다 |
| `ledger_entries` | 여러 개 생긴다 | 실제 debit/credit 기록을 남긴다 |

단, 고객이나 가맹점이 처음으로 특정 통화를 사용한다면 `ledger_accounts` row가 새로 생길 수 있다.

예를 들어 같은 고객이 기존에는 USDC만 사용하다가 KRW 스테이블코인을 새로 사용하면 아래처럼 별도 계정 row가 필요할 수 있다.

```plaintext
customer_1 / CUSTOMER / USDC
customer_1 / CUSTOMER / KRWS
```

## 4. Settlement 상태

Settlement는 확정된 Ledger 기록을 기준으로 "지급 가능한 금액을 계산하고 묶는 과정"이다.

대상은 꼭 가맹점일 필요는 없다. StablePay 결제 게이트웨이 관점에서는 가맹점 정산이 가장 대표적이지만, 서비스 모델에 따라 개인, 파트너, 셀러, 크리에이터, 기관이 정산 대상이 될 수 있다.

| 상태 | 의미 |
| --- | --- |
| `DRAFT` | 정산 후보를 계산 중인 상태 |
| `READY` | 지급 가능 금액이 계산되어 검토 가능한 상태 |
| `APPROVED` | 내부 정책상 정산을 진행해도 된다고 승인된 상태 |
| `PAID` | 실제 지급 또는 출금까지 완료된 상태 |
| `FAILED` | 정산 중 오류가 발생한 상태 |

정산 상태 전이는 보통 아래처럼 흐른다.

```plaintext
DRAFT
  |
  | 정산 대상 ledger entry 수집
  v
READY
  |
  | 운영 정책 검토 및 승인
  v
APPROVED
  |
  | 지급/출금 처리 완료
  v
PAID

오류 발생 시 FAILED
```

여기서 "지급 가능한 금액"이라는 말은 아무 금액이나 바로 보낸다는 뜻이 아니다.

정산 대상이 되려면 최소한 다음 조건을 확인해야 한다.

| 확인 항목 | 이유 |
| --- | --- |
| Payment가 확정되었는가 | 아직 finality가 부족한 결제를 지급하면 위험하다 |
| Ledger entry가 중복 생성되지 않았는가 | 같은 입금이 두 번 반영되면 손실이 생긴다 |
| 환불/취소/분쟁 대상이 아닌가 | 지급 후 되돌리기 어려울 수 있다 |
| 수수료 정책이 반영되었는가 | 플랫폼 수수료와 지급액을 분리해야 한다 |
| 최소 지급 금액이나 지급 주기 조건을 만족하는가 | 운영 정책에 따라 묶어서 지급할 수 있다 |

## 5. Withdrawal 상태

Withdrawal은 우리 시스템에서 외부 주소로 자산을 내보내는 흐름이다.

Deposit은 이미 온체인에서 발생한 일을 감지하는 쪽에 가깝고, Withdrawal은 우리가 transaction을 만들고 서명하고 전송하는 쪽에 가깝다.

| 상태 | 의미 |
| --- | --- |
| `REQUESTED` | 출금 요청이 생성된 상태 |
| `APPROVED` | 내부 정책상 출금이 가능하다고 승인된 상태 |
| `SIGNED` | 블록체인에 전송할 transaction에 서명이 완료된 상태 |
| `BROADCASTED` | 서명된 transaction을 블록체인 네트워크에 전송한 상태 |
| `CONFIRMED` | 온체인에서 충분히 확정된 상태 |
| `FAILED` | 주소 오류, 잔액 부족, 서명 실패, 전송 실패 등으로 실패한 상태 |

Withdrawal 상태 전이는 아래처럼 이해하면 된다.

```plaintext
REQUESTED
  |
  | 주소, 잔액, 한도, 보안 정책 검토
  v
APPROVED
  |
  | Rust Signer 등 서명 경계에서 tx 서명
  v
SIGNED
  |
  | Go Backend가 signed tx를 network에 broadcast
  v
BROADCASTED
  |
  | confirmation/finality 확보
  v
CONFIRMED

어느 단계에서든 실패 시 FAILED
```

여기서 `SIGNED`는 Ledger에 기록했다는 뜻이 아니다.

`SIGNED`는 블록체인 네트워크에 제출할 transaction payload에 개인키 기반 서명이 붙었다는 뜻이다.

| 구분 | 의미 |
| --- | --- |
| Ledger 기록 | 내부 DB에 돈의 이동을 장부로 남김 |
| Transaction signing | 블록체인에 제출할 transaction에 암호학적 서명을 붙임 |

## 6. 상태와 계정 버킷을 섞으면 생기는 문제

상태와 계정 타입을 섞어서 생각하면 아래와 같은 혼란이 생긴다.

| 혼란 | 올바른 구분 |
| --- | --- |
| `MERCHANT_PENDING`도 pending이니까 Payment 상태 아닌가? | 아니다. Ledger 계정 타입이다 |
| Payment가 `FINALIZED`이면 돈이 바로 지급된 것 아닌가? | 아니다. Ledger 기록과 Settlement가 남아 있다 |
| Settlement가 `PAID`이면 온체인 출금도 무조건 완료인가? | 설계에 따라 Withdrawal `CONFIRMED`와 연결해서 판단해야 한다 |
| `SIGNED`이면 원장 기록이 끝난 것인가? | 아니다. 출금 tx 서명이 완료된 것이다 |

이 프로젝트에서는 다음처럼 생각하면 된다.

```plaintext
Payment 상태 = 결제가 어디까지 확인되었는가
Ledger 계정 타입 = 돈을 어느 장부 칸에 기록할 것인가
Settlement 상태 = 지급 가능한 금액 묶음이 어디까지 처리되었는가
Withdrawal 상태 = 외부 주소로 보내는 tx가 어디까지 처리되었는가
```

## 7. 10 USDC 결제 예시

아래 예시는 앞으로 구현할 수 있는 목표 흐름이다.

현재 Day16 기준 코드/테이블 초안에는 `CUSTOMER`, `MERCHANT_PENDING`, `PLATFORM_FEE` 정도의 Ledger 기본 구조를 먼저 다룬다. `MERCHANT_SETTLEMENT_RESERVED`, `WITHDRAWAL_PENDING` 같은 계정 타입은 이후 설계 후보로 생각하면 된다.

```plaintext
1. 사용자가 10 USDC 결제
   Payment: PENDING

2. Event Indexer가 온체인 tx/event 감지
   Payment: ONCHAIN_DETECTED

3. confirmation/finality 확보
   Payment: FINALIZED

4. Ledger 기록 생성
   CUSTOMER          DEBIT  10.0 USDC
   MERCHANT_PENDING  CREDIT  9.8 USDC
   PLATFORM_FEE      CREDIT  0.2 USDC

5. 정산 대상 계산
   Settlement: DRAFT -> READY

6. 내부 정책 승인
   Settlement: APPROVED

7. 외부 지급/출금 진행
   Withdrawal: REQUESTED -> APPROVED -> SIGNED -> BROADCASTED -> CONFIRMED

8. 정산 완료
   Settlement: PAID
```

## 8. 구현할 때의 기준

상태 전이를 코드로 구현할 때는 보통 아래 기준을 둔다.

| 기준 | 설명 |
| --- | --- |
| 허용된 전이만 가능하게 한다 | 예: `PENDING -> FINALIZED`를 바로 허용하지 않는다 |
| 실패 상태를 명확히 둔다 | 검증 실패, 전송 실패, finality 실패를 구분할 수 있어야 한다 |
| 상태 변경의 이유를 남긴다 | 나중에 장애 복구와 운영 확인이 가능해야 한다 |
| 원장 기록은 멱등하게 만든다 | 같은 event를 여러 번 읽어도 Ledger entry가 중복되면 안 된다 |
| 상태와 금액 기록을 분리한다 | 상태는 진행 흐름, Ledger는 돈의 이동 기록이다 |

## 9. 복습 질문

1. `Payment.PENDING`과 `AccountType.MERCHANT_PENDING`은 무엇이 다른가?
2. Payment가 `FINALIZED`가 되면 왜 바로 정산 완료라고 말할 수 없는가?
3. `ledger_accounts` row는 결제마다 새로 생기는가?
4. `ledger_transactions.reference_id`, `reference_type`은 왜 필요한가?
5. Withdrawal의 `SIGNED`는 Ledger 기록과 같은 일인가?
6. Settlement 대상이 꼭 가맹점이어야 하는가?
7. 상태 전이 규칙을 코드로 막아야 하는 이유는 무엇인가?
