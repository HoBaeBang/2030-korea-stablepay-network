# Day 11 기초학습 - 프로젝트 중심 학습 리듬 전환과 Ledger 준비

관련 Jira: [SPN-28](https://aslan0.atlassian.net/browse/SPN-28)

Day11은 Backend Core를 길게 복습하는 날이 아니라, 앞으로의 학습 방식을 바꾸는 기준점입니다.

지금까지의 문제는 다음이었습니다.

```text
문서 작성량이 많아졌지만, 실제 프로젝트 코드에서 얻는 지식은 충분히 쌓이지 않았다.
```

따라서 Day11부터는 다음 원칙으로 진행합니다.

```text
작은 코드 작업 하나
  -> 그 코드가 왜 필요한지 이해
  -> 짧은 산출물 작성
  -> Codex 리뷰
  -> Git/Jira/Confluence 정리
```

## 오늘의 큰 그림

![Day11 Backend Core에서 Ledger로 넘어가는 준비도](../../../confluence/diagrams/spn28-day11-ledger-readiness.png)

Day11의 핵심은 “Day8~10을 완벽히 외운다”가 아닙니다.

오늘의 핵심은 아래 문장입니다.

```text
Payment 상태 전이 테스트를 이해한 뒤,
다음 Ledger 구현을 시작할 수 있을 만큼만 Backend Core를 정리한다.
```

## Day11에서 하지 않는 것

아래는 오늘 하지 않습니다.

```text
Ledger 전체 설계 완성
Ledger 코드 구현
긴 체크리스트 채우기
모든 도메인 용어 암기
빈칸 많은 산출물 작성
```

## Day11에서 하는 것

아래만 확실히 합니다.

```text
1. Day10 payment 테스트가 무엇을 막는지 설명한다.
2. 테스트와 로그의 차이를 설명한다.
3. Ledger가 왜 다음 구현 대상인지 설명한다.
4. Ledger 첫 구현을 파일 단위로 쪼갠다.
5. 다음 작업을 시작하기 전에 남은 위험을 정리한다.
```

## 출퇴근 학습

출퇴근 시간에는 아래 세 문장만 잡으면 됩니다.

```text
Payment는 결제의 상태를 말한다.
Ledger는 돈의 이동 기록을 말한다.
Test는 상태와 돈의 규칙이 깨지지 않게 고정하는 장치다.
```

## 퇴근 후 작업

퇴근 후에는 긴 문서 작성이 아니라 아래 순서로 진행합니다.

```text
1. internal/payment/service_test.go를 연다.
2. Day10에서 추가한 테스트를 다시 읽는다.
3. go test ./internal/payment 를 실행한다.
4. go test ./... 를 실행한다.
5. Day11 실습산출물의 핵심 5문항만 작성한다.
```

## 완료 기준

- [ ] Day10 payment 테스트가 막는 버그를 설명할 수 있다.
- [ ] 테스트와 로그의 차이를 설명할 수 있다.
- [ ] Payment와 Ledger의 차이를 설명할 수 있다.
- [ ] Ledger 첫 구현 후보를 파일 단위로 말할 수 있다.
- [ ] `go test ./...` 결과를 확인했다.
- [ ] Day11 실습산출물의 핵심 5문항을 작성했다.
