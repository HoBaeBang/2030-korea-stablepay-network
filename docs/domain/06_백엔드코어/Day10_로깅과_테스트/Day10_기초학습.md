# Day 10 기초학습 - 로깅과 테스트 패턴

관련 Jira: [SPN-27](https://aslan0.atlassian.net/browse/SPN-27)

Day 10의 목표는 로그와 테스트 패턴을 정리하는 것입니다.

Phase 2에서는 돈의 이동, 상태 변경, 온체인 이벤트 처리 같은 작업이 많아집니다.

이때 로그와 테스트가 없으면 문제가 생겼을 때 어디서 왜 틀어졌는지 알기 어렵습니다.

## 오늘의 큰 그림

![Day10 로깅과 테스트 피드백 루프](../../../confluence/diagrams/spn27-day10-logging-test-feedback.png)

로그와 테스트는 역할이 다릅니다.

테스트는 개발 중에 “이 규칙이 앞으로도 깨지지 않아야 한다”는 약속을 자동으로 확인합니다.

로그는 운영 중에 “실제로 어떤 일이 일어났는지”를 추적하는 기록입니다.

둘 중 하나만 있으면 부족합니다.

| 상황 | 테스트만 있고 로그가 없을 때 | 로그만 있고 테스트가 없을 때 |
| --- | --- | --- |
| payment 상태 전이 | 코드 변경 전에는 안전하지만 운영 장애 추적이 어렵다 | 장애 추적은 가능하지만 같은 버그 재발을 막기 어렵다 |
| 중복 이벤트 처리 | 중복 방어 규칙은 검증되지만 실제 몇 번 무시됐는지 모른다 | 실제 중복은 보이지만 코드가 항상 안전한지 보장하기 어렵다 |
| Ledger 기록 | 합계 0 규칙은 검증되지만 운영 중 어떤 거래가 생성됐는지 모른다 | 운영 기록은 남지만 잘못된 원장 생성 방어가 약하다 |

## 오늘의 목표

1. 어떤 상황에서 로그가 필요한지 설명할 수 있다.
2. 로그와 에러 응답의 차이를 이해한다.
3. Go 테스트의 기본 구조를 다시 복습한다.
4. 한글 subtest로 테스트 케이스를 읽기 좋게 작성하는 방식을 정리한다.
5. Backend Core 이후 Ledger 테스트 패턴을 준비한다.

## 오늘 꼭 잡아야 하는 문장

```text
로그는 운영 중인 시스템의 기억이고,
테스트는 코드가 지켜야 할 약속이다.
```

Day10을 지나면 “테스트를 작성했다”에서 끝나지 않고, 다음 질문에 답할 수 있어야 합니다.

```text
이 기능이 운영 중 실패하면 어떤 로그로 추적할 것인가?
이 기능이 코드 변경으로 깨지면 어떤 테스트가 잡아줄 것인가?
로그에는 어떤 식별자를 남기고 어떤 비밀값은 남기지 않을 것인가?
```

## 완료 기준

- [ ] 로그가 필요한 이벤트 후보를 작성했다.
- [ ] 테스트 패턴을 given/when/then으로 정리했다.
- [ ] 한글 subtest 예시를 작성했다.
- [ ] 실습산출물을 작성했다.
- [ ] 검증문제를 풀었다.
