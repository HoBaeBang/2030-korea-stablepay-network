## 1. main.go 실행 순서

아래 순서를 실제 `cmd/api/main.go`를 읽고 프로젝트 기준으로 채운다.

작성 방법:

```text
cmd/api/main.go를 위에서 아래로 읽으면서 "서버가 실행되기 전에 어떤 준비를 하는지"를 적는다.
코드에서 함수 호출이나 변수 생성이 보이면, 그 동작이 어떤 준비 단계인지 내 말로 바꿔 적는다.
```

| 순서 | 코드에서 확인한 동작 | 관련 파일/함수 | 내가 이해한 의미 |
| --- | --- | --- | --- |
| 1 | 설정값을 읽는다 | `cfg := config.Load()` | `config` 패키지를 통해 서버 실행에 필요한 설정값을 먼저 불러온다. `cfg`는 config 값을 담은 변수다. |
| 2 | DB 연결을 만든다 | `database.Open(ctx, cfg.DatabaseURL)` | HTTP 서버가 요청을 받기 전에 데이터베이스에 연결할 수 있는지 먼저 확인한다. 실패하면 서버를 시작하지 않는다. |
| 3 | HTTP 요청 분배 객체를 만든다 | `mux := http.NewServeMux()` | `mux`는 요청 경로와 method에 맞는 handler로 요청을 보내는 역할을 한다. |
| 4 | route를 등록한다 | `httpapi.RegisterHealthRoutes`, `RegisterMerchantRoutes`, `RegisterInvoiceRoutes`, `RegisterPaymentRoutes` | health, merchant, invoice, payment API가 HTTP 서버에 연결된다. |
| 5 | HTTP 서버를 만들고 실행한다 | `&http.Server{...}`, `server.ListenAndServe()` | 설정에서 가져온 `cfg.Port`로 서버를 실행하고, 등록된 route들이 요청을 처리할 수 있게 한다. |

관찰 메모:

```text
main.go에서 가장 먼저 눈에 들어온 코드:
cfg := config.Load()
이 코드를 통해 main.go가 직접 환경 변수를 읽지 않고 config 패키지에서 설정값을 받아온다는 점이 눈에 들어왔다.

아직 이해가 안 되는 코드:
처음에는 &http.Server{...}에서 &가 왜 붙는지와 mux가 정확히 무엇인지 헷갈렸다.
지금은 &는 Server 구조체 값의 주소를 가져와 포인터로 사용하는 것이고, mux는 요청을 적절한 handler로 분배하는 객체라고 이해했다.

Java/Spring Boot와 비교했을 때 비슷하다고 느낀 부분:
Spring Boot에서 application.yml 또는 application.properties 값을 읽어서 서버 포트, DB 설정을 구성하는 것과 비슷하다.
다만 Go에서는 프레임워크가 자동으로 해주는 부분이 적어서, main.go에서 설정 로딩과 의존성 조립 순서가 더 명시적으로 보인다.
```

## 2. 현재 필요한 설정값

작성 방법:

```text
현재 main.go 또는 새로 만든 config.go에서 환경 변수로 읽는 값을 찾는다.
그 값이 없으면 서버가 실행될 수 있는지, 기본값으로 대체해도 되는지 판단한다.
```

| 설정 | 필수 여부 | 기본값 가능 여부 | 이유 |
| --- | --- | --- | --- |
| `PORT` | 선택 | 가능 | 로컬 개발에서는 `8080` 같은 기본 포트를 둘 수 있다. 포트가 비어 있다고 해서 데이터가 망가지지는 않는다. |
| `DATABASE_URL` | 필수 | 운영 환경에서는 불가능 | DB 연결 주소가 잘못되면 결제, invoice, payment 데이터를 잘못된 DB에 저장하거나 아예 저장하지 못한다. 운영에서는 기본값으로 대체하면 위험하다. |
| `LOG_LEVEL` | 선택 | 가능 | 아직 코드에는 없지만 운영 로그 수준을 조절하기 위한 값이다. 기본값을 `INFO` 정도로 둘 수 있다. |

판단 기준 메모:

```text
필수 설정이라고 생각한 이유:
설정이 없거나 잘못되었을 때 잘못된 외부 자원에 연결되거나, 서비스 데이터가 다른 환경에 저장될 수 있으면 필수 설정이라고 봐야 한다.

기본값을 둘 수 있다고 생각한 이유:
로컬 개발 편의성을 높이고, 값이 비어 있어도 서비스 정합성에 큰 문제가 없는 값은 기본값을 둘 수 있다.
예를 들어 PORT는 로컬에서 8080을 기본값으로 사용해도 괜찮다.

아직 판단이 어려운 설정:
Phase 2에서 추가될 Blockchain RPC, Signer URL 같은 설정은 모든 프로세스에서 필수인지, 특정 기능에서만 필수인지 아직 더 설계가 필요하다.
```

## 3. Phase 2에서 추가될 설정값

작성 방법:

```text
Phase 2에서 등장할 기능을 떠올리고, 그 기능이 외부 시스템 주소나 실행 기준을 필요로 하는지 생각한다.
정확한 이름을 몰라도 괜찮다. "블록체인 RPC 주소 같은 것"처럼 먼저 적어도 된다.
```

| 설정 | 필요한 기능 | 설명 |
| --- | --- | --- |
| `BLOCKCHAIN_RPC_URL` | Event Indexer, Withdrawal | 블록체인 노드와 통신하기 위한 RPC 주소다. 입금 이벤트 조회나 출금 transaction 전송에 필요하다. |
| `INDEXER_POLL_INTERVAL` | Event Indexer | 인덱서가 블록체인 이벤트를 몇 초마다 조회할지 정한다. |
| `FINALITY_CONFIRMATIONS` | Deposit | 입금을 최종 확정으로 볼 confirmation 기준을 정한다. |
| `SIGNER_BASE_URL` | Withdrawal Signer | Go 백엔드가 Rust signer 서비스에 서명 요청을 보낼 때 필요한 주소다. |
| `LOG_LEVEL` | Logging | 운영 환경에서 로그를 어느 수준까지 출력할지 정한다. |

추가로 필요할 것 같은 설정:

```text
CHAIN_ID 또는 NETWORK_NAME:
나중에 mainnet, testnet, localnet을 구분해야 할 수 있다.

SERVICE_MODE:
API 서버, indexer, worker처럼 같은 코드베이스에서 실행 역할이 나뉘면 필요할 수 있다.
```

## 4. config 구현 후보

작성 방법:

```text
Day9 실습가이드에서 만든 config 패키지를 기준으로 적는다.
실제로 만든 파일 경로, 구조체 이름, 함수 이름을 적고, 왜 그렇게 나눴는지 적는다.
```

```text
패키지 위치: internal/platform/config
구조체 이름: Config
함수 후보: Load()
```

고민할 점:

```text
config가 어디까지 책임져야 한다고 생각하는가?
환경 변수에서 값을 읽고, 기본값을 적용하고, 값이 비어 있으면 안 되는 설정을 검증하는 정도까지 책임지는 것이 좋다고 생각한다.
DB 연결 자체는 database 패키지의 책임이고, config는 DB 연결에 필요한 주소를 준비하는 책임만 가진다.

기본값 적용은 어디에서 하는 것이 좋다고 생각하는가?
config.go 안에서 하는 것이 좋다.
그래야 main.go가 "PORT가 비어 있으면 8080" 같은 세부 규칙을 몰라도 된다.

설정 누락 에러는 어떤 메시지로 보여주는 것이 좋다고 생각하는가?
어떤 설정이 누락되었는지 명확히 보여줘야 한다.
예: "DATABASE_URL is required" 또는 "database url is empty"
다만 현재 실습 코드에서는 `DATABASE_URL`에 로컬 기본값을 넣고 있으므로, 운영 환경에서는 이 정책을 더 엄격하게 바꿀 필요가 있다.
```

## 5. 설정 누락 처리 기준

작성 방법:

```text
설정값이 없을 때 서버를 바로 종료해야 하는지, 기본값으로 실행해도 되는지, 특정 기능을 사용할 때만 막아도 되는지 나눠 적는다.
```

| 구분 | 설정 후보 | 이유 |
| --- | --- | --- |
| 서버 시작 시 반드시 실패해야 하는 설정 | `DATABASE_URL` | 운영 환경에서 DB 주소가 없거나 잘못되면 데이터를 잘못 저장하거나 서비스가 정상 동작하지 않는다. |
| 기본값을 사용해도 되는 설정 | `PORT`, `LOG_LEVEL` | 로컬 개발에서는 기본 포트와 기본 로그 수준을 둘 수 있고, 값이 없어도 데이터 정합성에 직접적인 문제를 만들 가능성이 낮다. |
| 특정 기능 실행 시점에 검증해도 되는 설정 | `BLOCKCHAIN_RPC_URL`, `SIGNER_BASE_URL`, `INDEXER_POLL_INTERVAL` | API 서버만 실행할 때는 필요 없을 수 있지만, indexer나 withdrawal 기능을 실행할 때는 필수다. |

추가 메모:

```text
설정 누락 기준은 모든 프로세스에 동일하게 적용하면 안 될 것 같다.
API 서버, Event Indexer, Withdrawal Worker처럼 실행 역할이 달라지면 필수 설정도 달라질 수 있다.

지금 실습에서는 DATABASE_URL에 로컬 기본값이 있지만, 실제 운영 서비스라면 운영 DB 주소는 기본값 없이 명시적으로 주입받는 것이 안전하다.
```

## 6. 오늘 헷갈린 개념

작성 방법:

```text
단어만 적지 말고, "어떤 코드에서 왜 헷갈렸는지"까지 적는다.
```

```text
- 헷갈린 코드/개념:
  cfg := config.Load()
  왜 헷갈렸는지:
  처음에는 cfg가 무엇을 의미하는지 바로 와닿지 않았다.
  config의 줄임말로 자주 쓰는 변수명이며, Config 구조체 값을 담고 있다고 이해했다.

- 헷갈린 코드/개념:
  databaseURL 기본값
  왜 헷갈렸는지:
  로컬 개발에서는 편하지만 운영 환경에서는 기본값이 위험할 수 있다는 점이 헷갈렸다.
  기본값은 개발 편의와 운영 안전성 사이에서 다르게 판단해야 한다고 이해했다.

- 헷갈린 코드/개념:
  main.go와 config.go의 책임 분리
  왜 헷갈렸는지:
  단순히 os.Getenv를 옮긴 것처럼 보였지만, 실제로는 main.go가 실행 조립에 집중하고 config.go가 설정 로딩에 집중하도록 책임을 나누는 작업이었다.
```

```text
다만 어떤 설정에 기본값을 둘 수 있고 어떤 설정은 반드시 명시해야 하는지는 로컬 개발 환경과 운영 환경을 나누어 판단해야 한다.
```
