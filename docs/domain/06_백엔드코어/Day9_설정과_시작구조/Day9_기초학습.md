# Day 9 기초학습 - 설정 로딩과 애플리케이션 시작 구조

관련 Jira: [SPN-26](https://aslan0.atlassian.net/browse/SPN-26)

Day 9의 목표는 서버 실행에 필요한 설정을 한 곳에서 읽고, 애플리케이션 시작 구조를 이해하는 것입니다.

## 오늘의 큰 그림

![Day9 설정 로딩과 애플리케이션 시작 구조](../../../confluence/diagrams/spn26-day9-config-startup-flow.png)

Day9의 핵심은 `main.go`를 “모든 일을 하는 파일”이 아니라 “애플리케이션을 조립하는 진입점”으로 이해하는 것입니다.

Java Spring에서는 설정, Bean 생성, 의존성 주입을 프레임워크가 상당 부분 대신해줍니다. Go에서는 그 과정을 코드로 직접 드러내는 경우가 많습니다.

그래서 Go의 `main.go`를 읽을 때는 다음 순서로 봅니다.

| 순서 | 하는 일 | 왜 중요한가 |
| --- | --- | --- |
| 1 | 환경 변수 읽기 | 실행 환경마다 달라지는 값을 코드 밖에서 주입한다 |
| 2 | Config 구조체 생성 | 흩어진 설정값을 한 곳에 모은다 |
| 3 | DB 연결 | Repository가 사용할 외부 자원을 준비한다 |
| 4 | Repository/Service/Handler 조립 | 의존성 방향을 명확히 만든다 |
| 5 | Route 등록 | 외부 HTTP 요청이 어떤 handler로 들어갈지 연결한다 |
| 6 | Server 시작 | 준비가 끝난 뒤 요청을 받기 시작한다 |

## 왜 설정 구조가 필요한가?

처음에는 `PORT`, `DATABASE_URL` 정도만 필요합니다.

하지만 Phase 2가 진행되면 다음 설정이 추가됩니다.

```text
BLOCKCHAIN_RPC_URL
INDEXER_POLL_INTERVAL
FINALITY_CONFIRMATIONS
SIGNER_BASE_URL
LOG_LEVEL
```

이 값들이 코드 여러 곳에 흩어지면 변경하기 어렵고, 테스트도 어려워집니다.

특히 Phase 2에서는 설정이 단순 실행값이 아니라 안정성 정책으로 이어집니다.

| 설정 | 잘못되면 생기는 문제 |
| --- | --- |
| `DATABASE_URL` | 결제, 원장, 정산 데이터를 저장할 수 없다 |
| `BLOCKCHAIN_RPC_URL` | 온체인 이벤트를 읽거나 출금 transaction을 전송할 수 없다 |
| `FINALITY_CONFIRMATIONS` | 입금을 너무 빨리 확정하거나 너무 늦게 반영할 수 있다 |
| `SIGNER_BASE_URL` | Rust signer와 연결하지 못해 출금을 진행할 수 없다 |
| `LOG_LEVEL` | 장애 분석에 필요한 로그가 부족하거나 너무 많아질 수 있다 |

## 오늘의 목표

1. config가 무엇인지 설명할 수 있다.
2. 환경 변수와 코드 설정의 차이를 이해한다.
3. `main.go`가 어떤 순서로 서버를 시작하는지 설명할 수 있다.
4. Phase 2에 필요한 설정 후보를 정리한다.
5. 설정 누락 시 어떤 에러를 내야 하는지 판단한다.

## 오늘 꼭 잡아야 하는 문장

```text
Config는 실행 환경과 애플리케이션 코드를 연결하는 경계다.
```

이 문장을 다르게 말하면, Config는 “코드가 어디에서 실행되는지”를 코드 안으로 안전하게 전달하는 계약입니다.
