# 프로젝트 운영 메모리

이 문서는 `2030 KOREA StablePay Network`를 진행할 때 Codex가 먼저 확인해야 하는 장기 운영 기준입니다.

대화 컨텍스트가 길어지거나 새 세션이 시작되면, 이 문서를 먼저 읽고 현재 학습/구현 리듬을 복원합니다.

## 현재 학습자 상태

```text
사용자는 주니어 Java 백엔드 개발자다.
Go, Rust, 블록체인 결제 도메인, 자료구조/알고리즘을 함께 보강하고 있다.
Go 문법 중 포인터, receiver, interface, context, 테스트 패턴은 반복 설명이 필요하다.
영어 용어는 가능하면 한글 의미와 함께 설명한다.
```

## 프로젝트 목표

```text
1차 목표:
스테이블코인 결제 백엔드 시스템을 Go로 구현하고 포트폴리오화한다.

2차 목표:
Ledger, Settlement, Deposit, Withdrawal, Event Indexer, Wallet/Key Security를 이해하고 구현한다.

장기 목표:
Rust 기반 signer 또는 chain prototype까지 확장해 블록체인 기반 결제/토큰 직무로 전환 가능한 역량을 만든다.
```

## 학습 방식 원칙

```text
문서를 많이 만드는 것보다 실제 기능 흐름 하나를 구현하고 테스트하는 것이 우선이다.
```

Day18 이후부터는 진도를 너무 느리게 끌지 않기 위해 기본 리듬을 압축해서 운영한다.

앞으로의 기본 리듬:

```text
1. 오늘 완성할 기능 흐름 1개를 정한다.
2. 그 기능에 필요한 관련 개념과 코드 흐름을 함께 설명한다.
3. 사용자가 코드를 작성하거나 Codex가 함께 구현한다.
4. 산출물은 5개 이하의 핵심 질문만 작성한다.
5. Codex가 사용자의 답변을 리뷰하고 오해를 바로잡는다.
6. Git, Jira, Confluence를 같은 내용으로 정리한다.
```

## Day20 이후 압축 진행 리듬

Day19까지 완료한 뒤에는 기존 Day20~35 일정을 새 Day20~27의 8회 일정으로 통합한다.
압축은 단순히 하루에 문서 두 묶음을 처리하는 방식이 아니다. 서로 이어지는 기존 Day를 **새 Day 하나의 구현 목표, 문서, 산출물, Jira 티켓**으로 합치는 방식이다.

```text
새 Day20 = 기존 Day20
새 Day21 = 기존 Day21~22
새 Day22 = 기존 Day23~24
새 Day23 = 기존 Day25~26
새 Day24 = 기존 Day27~28
새 Day25 = 기존 Day29~30
새 Day26 = 기존 Day31~32
새 Day27 = 기존 Day33~35
```

따라서 Phase 2의 현재 계획은 Day27에서 종료한다.

```text
1. 새 Day 하나에서 서로 이어지는 기존 Day 1~3개를 하나의 기능 흐름으로 구현한다.
2. 문서는 기존 Day별로 나누지 않고 새 Day 기준의 구현가이드와 실습산출물만 만든다.
3. 문서 작성보다 코드 구현과 테스트를 먼저 끝낸다.
4. Git, Jira, Confluence 정리는 구현이 끝난 뒤 한 번에 정리한다.
5. 복습과 검증은 별도 하루로 분리하지 않고 각 새 Day의 마지막 단계에 포함한다.
   사용자가 특정 구간을 다시 확인하고 싶다고 요청할 때만 추가 회고 세션을 만든다.
6. 이해가 부족한 핵심 개념은 압축 대상에서 삭제하지 않고 구현가이드 안에서 충분히 설명한다.
7. 일정표의 기존 Day 번호보다 이 운영 메모리의 새 Day20~27 매핑을 우선한다.
```

하루 운영 흐름:

```text
출퇴근 시간:
- 오늘 만들 기능의 배경
- 왜 이 기능이 필요한지
- 어떤 파일을 건드리는지
- 오늘 헷갈릴 Go/DB 개념이 무엇인지

퇴근 후:
- 바로 코드 작업으로 들어간다.
- 기능 흐름 구현 -> 단위/통합 테스트 -> 핵심 복습 -> 산출물 작성 순서로 진행한다.

작업 종료 후:
- Codex 리뷰
- Git 커밋/푸시
- Jira 댓글/상태 업데이트
- Confluence 동기화
```

```text
실습가이드는 그 문서만 보고 실습할 수 있어야 한다.
실습산출물은 빈칸만 던지지 않는다.
각 항목마다 무엇을 보고 무엇을 적어야 하는지 알려준다.
검증문제는 질문과 답변가이드를 분리한다.
다이어그램이 이해를 돕는 경우 반드시 추가한다.
GitHub Markdown과 Confluence 내용은 서로 싱크를 맞춘다.
Confluence 문서도 GitHub 문서와 같은 상세도를 유지한다. Confluence에는 요약본만 올리지 않는다.
```

## Day 문서 구조 원칙

Day15~Day17까지는 기존 3문서 체계를 유지한다.

```text
DayXX_학습_및_실습가이드.md
DayXX_실습산출물.md
DayXX_검증문제_답변가이드.md
```

`DayXX_학습_및_실습가이드.md`는 기존의 기초학습, 개념학습, 실습가이드를 하나로 합친 문서다.

이 문서는 아래 내용을 한 흐름으로 포함해야 한다.

```text
오늘의 목표
왜 이 기능을 만드는가
핵심 배경 개념
핵심 용어
필요한 다이어그램
실습 전 확인 사항
수정/작성할 파일
작성할 코드 또는 SQL
실행할 명령
예상 결과
검증 방법
자주 만나는 오류
완성 기준
커밋 메시지
다음 작업 예고
```

`DayXX_실습산출물.md`은 사용자가 직접 답변하는 문서로 유지한다.

실습산출물에는 사용자가 답변을 작성한 뒤 비교할 수 있는 `정답/점검 가이드`를 하단에 포함한다.

실습산출물의 정답/점검 가이드는 정답이 바로 보이지 않도록 `<details>` / `<summary>` 접기 구조로 작성한다.

`DayXX_검증문제_답변가이드.md`는 복습용 문제와 답변을 분리해서 유지한다.

검증문제 답변가이드는 사용자가 먼저 작성할 수 있는 `내 답변 작성 공간`을 반드시 포함한다.

답변가이드는 그 아래에 두고, 정답은 접기 구조로 숨긴다.

검증문제 답변은 정답이 바로 보이지 않도록 GitHub Markdown의 `<details>` / `<summary>` 접기 구조를 기본으로 사용한다.

```md
### 1. 문제 제목

<details>
<summary>답변 보기</summary>

답변 내용

</details>
```

사용자가 먼저 문제를 풀 수 있도록 답변은 기본적으로 접혀 있어야 한다.

Confluence도 GitHub와 같은 문서 체계를 따른다. Day15~Day17은 기존 3문서, Day18 이후는 압축형 2문서를 기본으로 한다.

Day18 이후부터는 기본적으로 "압축형 2문서 체계"를 우선한다.

```text
DayXX_구현가이드.md
DayXX_실습산출물.md
```

즉, 출퇴근 예습 자료와 퇴근 후 실습 가이드를 하나의 `구현가이드`로 묶고, 산출물은 별도 문서 하나만 유지한다. 여러 기존 Day를 합치더라도 파일명에는 범위를 붙이지 않고 새 Day 번호 하나만 사용한다.

핵심 원칙:

```text
1. 출퇴근용 자료는 없애지 않는다.
2. 다만 "오늘 구현할 코드 중심 예습 자료" 형태로 합친다.
3. 검증문제는 별도 문서로 무조건 분리하지 않는다.
4. 필요한 경우 구현가이드 하단에 복습 질문과 접기형 답변 가이드를 함께 둔다.
5. 실습에 필요한 전체 코드와 설명은 절대 줄이지 않는다.
```

`구현가이드` 안에는 아래 흐름이 함께 들어가야 한다.

```text
오늘 만들 것
왜 필요한가
기존 코드 어디에 붙는가
출퇴근 예습 포인트
핵심 용어
핵심 다이어그램
확인할 기존 파일
수정/생성할 파일
Step별 실습 절차
최종 완성본 전체
실행 명령
검증 방법
자주 만나는 오류
커밋 메시지
다음 작업 예고
```

## Confluence Day 문서 구조 원칙

Confluence에서도 GitHub 문서 구조와 같은 흐름을 유지한다.

Confluence의 폴더 페이지, Day 메인 페이지, 하위 문서 제목은 GitHub `docs` 구조를 기준으로 맞춘다.

특히 `docs/domain` 아래의 도메인별 폴더 구조는 Confluence에서도 동일한 계층으로 관리한다.

`앞으로의 일정` 문서는 일정/로드맵 문서일 뿐, Day별 학습 문서의 부모가 되면 안 된다.

Ledger Core Day 문서는 `domain > 07_Ledger_Core` 하위에 Day별 메인 페이지를 둔다. 각 Day 메인 페이지 아래에는 해당 시기의 문서 체계에 맞춰 Day15~Day17은 3개, Day18 이후는 2개 문서만 둔다.

```text
domain
  07_Ledger_Core
    00_Ledger_Core_앞으로의_일정.md
    Day15_Ledger_Service_균형검증_구현
      Day15_학습_및_실습가이드.md
      Day15_실습산출물.md
      Day15_검증문제_답변가이드.md
    Day16_Ledger_DB_Migration_작성
      Day16_학습_및_실습가이드.md
      Day16_실습산출물.md
      Day16_검증문제_답변가이드.md
    Day17_Ledger_Repository_초안
      Day17_학습_및_실습가이드.md
      Day17_실습산출물.md
      Day17_검증문제_답변가이드.md
```

상위 문서 제목은 가능하면 문서가 설명하는 도메인 내용을 먼저 보여준다.

예:

```text
Day17_Ledger_Repository_초안              -> Ledger Repository 초안
Day16_Ledger_DB_Migration_작성           -> Ledger DB Migration 작성
Day15_Ledger_Service_균형검증_구현        -> Ledger Service 균형 검증 구현
```

다만 하위 문서는 학습 순서 추적이 중요하므로 `DayXX_학습_및_실습가이드.md`처럼 Day 번호를 유지할 수 있다.

기존 5문서 체계의 `개념학습.md`, `실습가이드.md` 안내 문서가 남아 있으면 Day 메인 페이지 아래에 두지 않는다.

삭제하지 않고 `archive_구버전문서` 아래로 이동해 과거 링크 호환과 내용 보존을 유지한다.

Day18 이후 압축형 문서도 Confluence에서 Git과 같은 구조로 맞춘다.

예:

```text
domain
  07_Ledger_Core
    Day21_Payment와_Ledger_연결_및_회고
      Day21_구현가이드.md
      Day21_실습산출물.md
```

즉 Day18 이후에는 Confluence도 기본적으로 2문서 체계를 따른다.

## 다이어그램 이미지 원칙

```text
다이어그램이나 이미지가 필요한 경우 GitHub에는 원본 SVG와 표시용 PNG를 함께 남긴다.
Confluence 표시용 이미지는 PNG를 우선 사용한다.
사용자가 다크 테마 환경에서 주로 확인하므로 새 다이어그램은 다크 테마 PNG를 기본으로 만든다.
배경만 어둡게 하지 말고 텍스트, 선, 박스, 강조색까지 충분한 대비가 나도록 조정한다.
SVG는 일부 환경에서 미리보기 문제가 생길 수 있으므로 Confluence에는 PNG 링크를 우선 연결한다.
Confluence 본문에 이미지를 넣을 때는 GitHub 상대경로를 그대로 쓰지 않는다.
Confluence는 `../../../confluence/...` 같은 상대경로를 해석하지 못하므로 `https://raw.githubusercontent.com/...` 형태의 절대 URL로 변환한다.
```

## 실습 코드 제공 원칙

```text
실습가이드는 코드 조각만 제공하면 안 된다.
사용자가 그 문서만 보고 실습할 수 있도록, 수정하거나 새로 작성할 파일마다 "최종 완성본 전체"를 제공한다.
문서가 통합 문서로 바뀌더라도 실습 내용의 밀도는 절대 줄이면 안 된다.
```

필수 규칙:

```text
1. Go 코드 실습이면 관련 파일별 전체 완성본을 제공한다.
   예: ledger.go, service.go, service_test.go

2. SQL migration 실습이면 up.sql과 down.sql의 전체 완성본을 제공한다.

3. 일부 코드 한두 줄만 보여주는 경우에도, 반드시 뒤쪽에 파일 전체 기준의 최종 완성본을 다시 제공한다.

4. 현재 파일과 완성본의 차이를 사용자가 비교할 수 있도록 "현재 확인할 파일"과 "최종 완성본 전체"를 분리해서 작성한다.

5. 실습자가 전체 서비스 흐름을 이미 알고 있다고 가정하지 않는다.
   코드가 어느 파일에 들어가고, 왜 그 파일에 들어가는지 함께 설명한다.

6. 완성본 전체가 길어지더라도 생략하지 않는다.
   실습 문서는 짧은 힌트 문서가 아니라 실제 작업을 끝낼 수 있는 가이드여야 한다.

7. Day13 실습가이드의 형식을 기준으로 유지한다.
   사전 조건 확인 -> 오늘 만들 파일 위치 -> Step별 코드/SQL 작성 -> 코드/SQL 해석 -> 실행 -> 오류 대응 -> 산출물 -> 커밋 메시지 흐름을 지킨다.

8. 통합 문서 안에서도 실습 파트는 반드시 Step 단위로 작성한다.
   출퇴근 학습 설명과 퇴근 후 실습 절차가 섞여서 흐려지면 안 된다.

9. 확인만 하는 파일은 전체 코드를 첨부하지 않는다.
   확인해야 할 타입, 메서드, SQL 일부만 발췌하고 확인 포인트를 함께 적는다.

10. 수정하거나 새로 생성하는 파일은 작성할 코드를 먼저 제시하고, 그다음에 코드의 역할과 문법을 설명한다.
    각 Step은 `파일 위치 -> 작성할 코드 전체 -> 코드 해설 -> 실행 및 검증` 순서를 지킨다.
    전체 완성본은 `<details>` / `<summary>` 접기 구조로 넣어 가독성을 유지한다.

11. 접기 제목은 무엇을 펼치는지 명확하게 쓴다.
    예: `service_test.go 최종 완성본 전체 보기`, `up.sql 최종 완성본 전체 보기`

12. 새 테이블, 새 패키지, 새 파일을 처음 소개할 때도 상세 설명보다 작성할 코드나 SQL을 먼저 제시한다.
    다만 코드를 잘못된 파일에 작성하지 않도록 파일 위치와 오늘 만드는 대상은 코드 앞에서 한두 문장으로만 안내한다.
    특히 DB 테이블은 SQL 전체를 먼저 보여준 다음 아래 내용을 순서대로 해설한다.
    - 이 테이블이 답하는 질문
    - 이 테이블이 저장하는 도메인 개념
    - 주요 컬럼의 의미
    - 다른 테이블과의 관계
    - 실제 결제 예시에서 어떤 row가 생기는지
    - 왜 이 순서로 생성해야 하는지

13. 테이블 이름만 나열하지 않는다.
    `ledger_accounts`, `ledger_transactions`, `ledger_entries`처럼 비슷한 이름이 여러 개 나오면
    SQL을 제시한 직후 각 테이블의 책임 차이를 표와 예시로 분리해서 설명한다.

14. 구현 가이드의 코드 설명 순서는 항상 아래 형식을 사용한다.

    ```text
    작성 대상과 파일 위치를 짧게 안내
      -> 작성할 코드 또는 SQL 제시
      -> 코드 블록 바로 아래에서 줄과 블록 단위로 해설
      -> 이 코드가 전체 요청 흐름에서 담당하는 역할 설명
      -> 실행 명령과 예상 결과
      -> 검증 방법과 자주 발생하는 오류
    ```

    설명을 모두 읽어야만 작성할 코드를 찾을 수 있는 구성은 사용하지 않는다.
    사용자는 먼저 코드를 작성하고, 이어지는 해설을 보면서 자신이 작성한 코드를 이해할 수 있어야 한다.

15. 프로젝트 인프라 실행 명령은 실제 저장소 구성을 기준으로 작성한다.
    StablePay의 PostgreSQL은 `docker-compose.yml`의 `postgres` 서비스를 사용하므로, 마이그레이션과 DB 확인 명령은 로컬 `psql`이 아니라 아래 형태를 기본으로 한다.

    ```bash
    docker compose exec -T postgres psql -U stablepay -d stablepay < migrations/파일명.sql
    docker compose exec -T postgres psql -U stablepay -d stablepay -c "\\dt"
    ```

    가이드를 작성하기 전에 `docker-compose.yml`, 실행 위치, 서비스 이름을 확인한다.
    사용자가 별도로 로컬 PostgreSQL 도구를 설치해야만 실행되는 명령을 기본 절차로 제시하지 않는다.

16. 학습용 Go 코드의 구조체 리터럴은 필드를 한 줄에 몰아 쓰지 않는다.
    `Entry`, `Transaction`, `Account`, 요청/응답 구조체처럼 필드가 여러 개인 값은 아래처럼 필드별 여러 줄 형식으로 작성한다.

    ```go
    entries := []Entry{
        {
            ID:            "led_entry_1",
            TransactionID: tx.ID,
            AccountID:     "acct_customer_1",
            Direction:     EntryDirectionDebit,
            Amount:        10_000_000,
            Currency:      "USDC",
        },
    }
    ```

    한 줄 구조체 리터럴은 필드 추가 여부를 놓치기 쉽고 학습자가 기존 예제와 차이를 비교하기 어렵기 때문에 사용하지 않는다.
    짧은 빈 구조체나 필드가 하나뿐인 단순 값만 예외로 허용한다.
```

문서 섹션 제목은 가능하면 아래 표현을 사용한다.

```text

```

## Go 테스트 작성 원칙

```text
테스트 중복을 줄이기 위해 service, repository, context를 패키지 전역 변수로 공유하지 않는다.
테스트끼리 상태가 섞이지 않도록 각 테스트 케이스는 독립적인 준비 과정을 가진다.
중복되는 테스트 준비 코드는 newTestService(t), newTestRepository(t) 같은 helper 함수로 줄인다.
t.Helper()를 helper 함수 안에서 호출해 실패 위치가 실제 테스트 케이스로 보이게 한다.
context.Background()도 반복이 많아지면 helper에서 함께 반환할 수 있다.
나중에 Service가 Repository, fake DB, time function, config를 갖게 되면 helper만 확장한다.
```

권장 형태:

```go
func newTestService(t *testing.T) (*Service, context.Context) {
	t.Helper()

	return NewService(), context.Background()
}
```

Repository가 붙은 뒤의 권장 형태:

```go
func newTestService(t *testing.T) (*Service, *fakeRepository) {
	t.Helper()

	repo := &fakeRepository{}
	svc := NewService(repo)

	return svc, repo
}
```
