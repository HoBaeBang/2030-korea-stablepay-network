# 블록체인 개발자 추가학습 로드맵

이 문서는 `2030 KOREA StablePay Network`를 실무형 블록체인 백엔드 포트폴리오로 확장하기 위해 추가로 학습해야 할 기술 요소와 핵심 개념을 정리한다.

목표는 기술 이름을 많이 나열하는 것이 아니라, 각 기술을 StablePay의 어떤 문제를 해결하기 위해 배우는지 분명히 하는 것이다.

## 먼저 가져갈 기준

현재 프로젝트의 중심은 Go 백엔드, PostgreSQL, Ledger, Settlement다.

이 기반 위에 Kafka, Redis, RPC Proxy, Kubernetes, Rust 등을 붙이는 이유는 다음 질문에 답하기 위해서다.

1. 블록체인 이벤트가 많이 들어와도 안정적으로 처리할 수 있는가?
2. 같은 이벤트를 여러 번 읽어도 돈이 중복 반영되지 않는가?
3. RPC 노드가 느리거나 실패해도 서비스를 계속 운영할 수 있는가?
4. 장애가 발생했을 때 어디서 깨졌는지 추적할 수 있는가?
5. 개인키와 서명 권한을 안전하게 분리할 수 있는가?
6. 실제 운영 환경에서 배포, 확장, 복구가 가능한 구조인가?

## 전체 학습 지도

| 우선순위 | 학습 요소 | 왜 필요한가 | StablePay에서 연결되는 기능 |
| --- | --- | --- | --- |
| 1 | Go 심화 | 현재 백엔드의 주 언어이며 대부분의 도메인 로직을 담당한다 | API, Service, Repository, Indexer, Worker |
| 1 | PostgreSQL과 SQL | Ledger와 Settlement의 정합성을 DB transaction으로 보장해야 한다 | 원장 저장, 정산 묶음, 멱등성, 대사 |
| 1 | 블록체인 RPC | 실제 체인에서 블록, 트랜잭션, 로그를 읽어야 한다 | Ethereum Indexer, Deposit 감지 |
| 1 | Indexer 설계 | 온체인 이벤트를 안전하게 읽고 재처리해야 한다 | Checkpoint, Reorg 대응, Event Idempotency |
| 2 | Kafka | 수집과 처리를 분리하고 대량 이벤트를 안정적으로 처리해야 한다 | Indexer -> Processor 이벤트 파이프라인 |
| 2 | Redis | 빠른 공유 상태, 제한, 캐시, 짧은 잠금이 필요하다 | Rate Limit, RPC 캐시, 작업 중복 방지 보조 |
| 2 | RPC Proxy | 여러 RPC 노드를 안정적으로 사용해야 한다 | Health Check, Retry, Circuit Breaker |
| 2 | Observability | 장애 원인을 로그와 지표로 찾을 수 있어야 한다 | Metrics, Logs, Tracing, Dashboard |
| 3 | Docker와 Kubernetes | 실제 서비스처럼 배포하고 확장해야 한다 | API, Indexer, Worker 분리 배포 |
| 3 | Cloud와 CI/CD | 외부에서 재현 가능한 운영 환경이 필요하다 | AWS, GitHub Actions, ArgoCD |
| 3 | Security와 Key Management | 출금 서명과 개인키는 백엔드와 분리해야 한다 | Wallet, Rust Signer, Key Boundary |
| 3 | Rust | 서명기, 고성능 컴포넌트, 체인 코어 이해에 필요하다 | Rust Signer, Chain Prototype |
| 4 | Non-EVM | Ethereum 외 체인 구조를 이해해야 한다 | Solana Indexer, 멀티체인 추상화 |
| 4 | Smart Contract | 온체인 결제와 이벤트 발생 구조를 이해해야 한다 | ERC-20, Payment Contract, Event Log |
| 4 | 성능 테스트 | 포트폴리오를 수치로 증명해야 한다 | p95/p99, throughput, consumer lag |

## 1. Go 심화

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Interface | 구현체 교체, 테스트 더블, 경계 분리 | Service가 Repository나 FakeStore를 받는 구조 |
| Context | 취소, timeout, 요청 생명주기 전달 | DB, RPC, Worker 작업 중단 |
| Error Wrapping | `%w`, `errors.Is`, `errors.As` | 도메인 오류와 인프라 오류 구분 |
| Goroutine | 동시 작업 실행 | Indexer, Processor, Scheduler |
| Channel | 작업 전달과 종료 신호 | Worker queue, graceful shutdown |
| Mutex | 공유 상태 보호 | in-memory 상태, 테스트 도구 |
| Package 구조 | 도메인별 패키지 설계 | `internal/ledger`, `internal/settlement` |
| Testing | table test, helper, fake, integration test | 서비스/레포지토리 검증 |

### 특히 집중할 것

Go를 단순히 문법으로 배우는 것이 아니라, 백엔드 프로세스가 오래 실행되면서 외부 시스템과 통신할 때 어떻게 안전하게 멈추고, 재시도하고, 오류를 전달하는지를 봐야 한다.

### 결과물

```text
Go Service/Repository 테스트를 스스로 작성할 수 있다.
context timeout을 걸고 DB/RPC 요청을 취소할 수 있다.
interface를 이용해서 테스트 fake를 만들 수 있다.
```

## 2. PostgreSQL과 SQL

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Transaction | 여러 INSERT/UPDATE를 하나의 원자적 작업으로 묶기 | Ledger transaction + entries 저장 |
| Isolation Level | 동시에 처리되는 요청의 충돌 방식 | Settlement 상태 전이, 중복 처리 |
| Unique Constraint | 같은 키가 두 번 저장되지 않도록 제한 | idempotency key, processed event key |
| Index | 조회 성능 최적화 | settlement candidate 조회 |
| JOIN | 여러 테이블을 연결해서 읽기 | ledger_entries와 ledger_accounts 비교 |
| LEFT JOIN | 기준 데이터는 유지하고 매칭 실패를 찾기 | reconciliation에서 누락된 ledger entry 찾기 |
| Lock | 동시에 같은 row를 수정하지 못하게 제어 | settlement approve/process |
| Migration | DB schema 변경 이력 관리 | `migrations/*.sql` |

### 특히 집중할 것

블록체인 백엔드는 단순 CRUD보다 DB 정합성이 훨씬 중요하다. 돈이 한 번 더 반영되거나, 정산 상태만 바뀌고 item 저장이 실패하면 실제 장애가 된다.

### 결과물

```text
Ledger와 Settlement 저장을 하나의 transaction으로 처리할 수 있다.
중복 이벤트가 들어와도 unique constraint로 막을 수 있다.
JOIN 결과를 보고 어떤 데이터가 누락됐는지 찾을 수 있다.
```

## 3. 블록체인 RPC

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| RPC | 원격 노드에 요청을 보내 체인 데이터를 읽는 방식 | Ethereum RPC Client |
| Block | 트랜잭션 묶음과 height/hash | Indexer의 읽기 단위 |
| Transaction | 사용자가 체인에 제출한 작업 | Deposit/Withdrawal 추적 |
| Transaction Receipt | 트랜잭션 실행 결과 | 성공 여부, gas, logs 확인 |
| Event Log | 컨트랙트가 남긴 이벤트 기록 | ERC-20 Transfer 감지 |
| Topic | 이벤트 검색을 위한 indexed field | 특정 이벤트 필터링 |
| Confirmation | 블록 뒤에 추가로 쌓인 블록 수 | 확정성 정책 |
| Finality | 되돌릴 수 없다고 판단하는 상태 | 자산 반영 시점 |

### 특히 집중할 것

RPC는 단순 API 호출처럼 보이지만, 블록체인에서는 같은 트랜잭션을 여러 번 조회하거나, 노드마다 응답이 다르거나, 아직 확정되지 않은 블록을 볼 수 있다. 그래서 “읽었다”와 “반영해도 된다”를 분리해야 한다.

### 결과물

```text
특정 block number의 block을 조회할 수 있다.
transaction receipt에서 log를 읽을 수 있다.
ERC-20 Transfer 이벤트를 파싱할 수 있다.
```

## 4. Indexer 설계

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Polling | 주기적으로 새 블록을 읽는 방식 | Phase 3 초기 Indexer |
| Checkpoint | 마지막 처리 위치 저장 | 재시작 후 이어서 처리 |
| Event Key | 이벤트의 유일성 키 | `chain + tx_hash + log_index` |
| Idempotency | 같은 이벤트를 여러 번 처리해도 결과가 같게 만들기 | 중복 입금 방지 |
| Reorg | 블록이 다른 체인으로 대체되는 현상 | rollback/replay |
| Backfill | 과거 구간을 다시 읽는 작업 | 장애 복구, 누락 복구 |
| Raw Event 저장 | 원본 이벤트를 먼저 저장하는 패턴 | 재처리와 감사 추적 |

### 특히 집중할 것

Indexer는 “이벤트를 읽는 코드”가 아니라 “불완전한 외부 세계를 안전하게 내부 DB 상태로 바꾸는 코드”다.

### 결과물

```text
Indexer를 껐다 켜도 checkpoint부터 이어서 처리한다.
같은 tx_hash/log_index가 다시 들어와도 Ledger가 중복 생성되지 않는다.
누락된 구간을 backfill로 다시 처리할 수 있다.
```

## 5. Kafka

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Topic | 이벤트가 쌓이는 논리적 스트림 | `blockchain.events` |
| Partition | topic을 나눈 병렬 처리 단위 | 이벤트 처리량 확장 |
| Producer | 이벤트를 Kafka에 쓰는 주체 | Indexer |
| Consumer | 이벤트를 읽어 처리하는 주체 | Event Processor |
| Consumer Group | 여러 consumer가 작업을 나눠 처리하는 구조 | Processor scale-out |
| Offset | 어디까지 읽었는지 나타내는 위치 | 재시작과 재처리 |
| Commit | 처리 완료 위치 저장 | 중복/유실 제어 |
| At-least-once | 최소 한 번 처리 보장 | 중복 가능성 전제로 idempotency 필요 |
| Dead Letter Queue | 처리 실패 이벤트 격리 | 운영 장애 분석 |
| Consumer Lag | 처리 지연량 | 성능 지표 |

### StablePay에서 왜 필요한가

초기에는 Indexer가 DB에 직접 저장해도 된다. 하지만 이벤트가 많아지면 “체인에서 읽는 작업”과 “Ledger/Deposit/Payment로 반영하는 작업”을 분리해야 한다.

```text
Indexer -> Kafka topic -> Processor -> DB/Ledger
```

이 구조를 만들면 Processor가 잠시 죽어도 Kafka에 이벤트가 남아 있고, 다시 실행하면 이어서 처리할 수 있다.

### 결과물

```text
Indexer가 읽은 raw event를 Kafka topic에 발행한다.
Processor가 consumer group으로 이벤트를 읽는다.
Processor를 중단했다가 재시작해도 offset 기준으로 이어서 처리한다.
consumer lag을 측정한다.
```

## 6. Redis

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Key-Value | 짧은 데이터를 빠르게 저장 | RPC 응답 캐시 |
| TTL | 시간이 지나면 자동 삭제 | 임시 상태, rate limit counter |
| Atomic Operation | 단일 명령의 원자성 | 카운터 증가, 제한 |
| Distributed Lock | 여러 프로세스 중 하나만 작업 | scheduler 중복 실행 방지 |
| Rate Limit | 요청 횟수 제한 | API/RPC 호출 보호 |
| Cache Invalidation | 캐시 만료와 갱신 | RPC stale data 방지 |
| Pub/Sub | 간단한 메시지 방송 | 로컬 알림, 운영 이벤트 |

### StablePay에서 왜 필요한가

Redis는 메인 저장소가 아니다. 돈의 최종 기록은 PostgreSQL과 Ledger에 남아야 한다. Redis는 빠른 임시 상태를 다루는 보조 도구로 사용한다.

### 결과물

```text
RPC 호출 결과를 짧은 TTL로 캐시한다.
동일한 worker job이 여러 프로세스에서 동시에 실행되지 않도록 lock을 건다.
API 또는 RPC 요청에 rate limit을 적용한다.
```

## 7. RPC Proxy

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Upstream | 실제 RPC 노드 주소 | Alchemy, Infura, local node |
| Health Check | 노드가 정상인지 확인 | 장애 노드 제외 |
| Load Balancing | 여러 노드에 요청 분산 | 트래픽 안정화 |
| Retry | 실패 요청 재시도 | 일시 장애 복구 |
| Timeout | 오래 걸리는 요청 중단 | API/Worker 보호 |
| Circuit Breaker | 계속 실패하는 노드 차단 | 장애 확산 방지 |
| Rate Limit | 노드별 요청 제한 | RPC 비용과 제한 보호 |
| Fallback | 실패 시 다른 노드 사용 | 가용성 향상 |

### StablePay에서 왜 필요한가

블록체인 백엔드는 RPC 노드에 크게 의존한다. RPC가 느리면 Indexer가 밀리고, RPC가 틀린 데이터를 주면 입금 반영이 위험해진다.

### 결과물

```text
복수 RPC upstream을 등록한다.
실패율이 높은 upstream을 자동 제외한다.
요청 latency, error rate, retry count를 metric으로 남긴다.
```

## 8. Observability

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Structured Logging | key-value 형태 로그 | payment_id, tx_hash, batch_id 추적 |
| Metrics | 수치 기반 상태 관찰 | events/sec, error rate |
| Tracing | 요청 흐름 추적 | API -> Service -> DB/RPC |
| Prometheus | metric 수집 | Worker/Indexer 지표 |
| Grafana | dashboard | 운영 화면 |
| Alerting | 이상 상태 알림 | consumer lag, RPC 장애 |

### 특히 집중할 것

포트폴리오에서는 “동작한다”보다 “어떻게 관찰하고 복구하는가”가 더 강한 증거가 된다.

### 결과물

```text
Indexer 처리량과 실패 횟수를 metric으로 남긴다.
Settlement/Reconciliation mismatch count를 dashboard에 표시한다.
consumer lag이나 RPC error rate가 높으면 alert 조건을 문서화한다.
```

## 9. Docker와 Kubernetes

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Dockerfile | 애플리케이션 이미지 만들기 | API/Worker 이미지 |
| Docker Compose | 로컬 통합 실행 | PostgreSQL, Kafka, Redis |
| Pod | Kubernetes의 실행 단위 | API, Indexer, Processor |
| Deployment | 무중단에 가까운 배포와 replica 관리 | API scale-out |
| Job/CronJob | 일회성/주기 작업 | backfill, reconciliation |
| ConfigMap/Secret | 설정과 민감정보 분리 | DB URL, RPC key |
| Service | Pod 접근 경로 | 내부 통신 |
| Ingress | 외부 HTTP 접근 | public API |
| HPA | 부하에 따른 자동 확장 | Processor scale |

### StablePay에서 왜 필요한가

블록체인 백엔드는 API 하나만 있는 구조가 아니다. Indexer, Processor, Reconciliation worker, RPC Proxy가 서로 다른 실행 단위로 분리된다.

### 결과물

```text
Docker Compose로 전체 로컬 환경을 실행한다.
Kubernetes에서 API와 Worker를 별도 Deployment로 실행한다.
Processor replica 수를 늘렸을 때 처리량 변화를 측정한다.
```

## 11. Security와 Key Management

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Private Key | 절대 노출되면 안 되는 서명 키 | Withdrawal |
| Public Key/Address | 공개 가능한 식별자 | Wallet address |
| Signing | transaction에 권한 증명 추가 | Rust Signer |
| Key Isolation | 키를 API 서버와 분리 | 보안 경계 |
| Policy Check | 서명 전 내부 승인 검증 | withdrawal approved |
| Audit Log | 누가 어떤 서명을 요청했는지 기록 | 보안 감사 |
| HSM/KMS | 키를 안전하게 보관하는 시스템 | 장기 확장 |

### StablePay에서 왜 필요한가

입금은 체인에서 읽는 작업이지만, 출금은 우리가 돈을 보내는 작업이다. 따라서 출금은 서명 권한과 정책 검증이 핵심이다.

### 결과물

```text
Go Backend는 unsigned transaction과 정책 정보를 만든다.
Rust Signer는 정책 확인 후 signed transaction을 반환한다.
Go Backend 또는 Broadcaster가 signed transaction을 체인에 전송한다.
서명 요청과 결과가 audit log로 남는다.
```

## 12. Rust

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Ownership | 값의 소유권과 이동 | 메모리 안전성 |
| Borrowing | 참조와 수명 | 안전한 데이터 접근 |
| Result/Option | 에러와 빈 값 처리 | 서명 실패, 검증 실패 |
| Trait | 공통 동작 추상화 | chain별 signer |
| Async | 비동기 I/O | signer service |
| Serialization | JSON/bytes 변환 | signing request/response |
| Crypto Crate | secp256k1, ed25519 등 | Ethereum/Solana 서명 |

### StablePay에서 왜 필요한가

Rust는 전체 백엔드를 대체하기보다, 개인키와 서명처럼 작고 중요한 경계에 먼저 적용한다.

### 결과물

```text
Rust signer service가 signing request를 받는다.
정책 검증 후 transaction에 서명한다.
Go Backend와 HTTP 또는 gRPC로 통신한다.
```

## 13. Smart Contract

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| ERC-20 | 토큰 표준 | USDC Transfer |
| Contract | 체인 위에서 실행되는 프로그램 | 결제 컨트랙트 |
| Event | contract가 외부에 남기는 로그 | Indexer 입력 |
| ABI | contract 호출/이벤트 해석 규칙 | log parsing |
| Gas | 실행 비용 | transaction fee |
| Revert | contract 실행 실패 | 결제 실패 처리 |

### StablePay에서 왜 필요한가

USDC 결제 자체는 ERC-20 Transfer만으로도 이해할 수 있다. 하지만 결제 상태를 온체인 이벤트로 더 풍부하게 남기려면 간단한 Payment Contract를 이해해야 한다.

### 결과물

```text
ERC-20 Transfer log를 ABI 기준으로 해석한다.
간단한 PaymentFinalized 이벤트를 발생시키는 contract를 작성한다.
contract event를 Indexer가 읽어 Payment 상태와 연결한다.
```

## 14. Non-EVM과 Solana

### 배워야 할 개념

| 개념 | 학습할 내용 | 프로젝트 연결 |
| --- | --- | --- |
| Account Model | Solana의 계정 기반 상태 모델 | Ethereum과 구조 차이 |
| Instruction | transaction 내부 실행 명령 | 이벤트 해석 |
| Program | Solana의 smart contract | SPL Token |
| Signature | transaction 식별과 서명 | tx 추적 |
| Commitment | processed/confirmed/finalized | finality 판단 |
| SPL Token | Solana 토큰 표준 | USDC on Solana |

### StablePay에서 왜 필요한가

멀티체인을 하려면 Ethereum 코드를 복사하는 것이 아니라, 공통 개념과 체인별 구현을 분리해야 한다.

### 결과물

```text
Ethereum과 Solana 이벤트를 공통 RawEvent 형태로 변환한다.
체인별 finality 기준을 따로 둔다.
같은 Deposit API에서 chain을 구분해서 조회한다.
```

## 학습할 때 주의할 점

기술을 먼저 붙이지 않는다.

```text
문제 확인
-> 최소 구현
-> 테스트
-> 장애 케이스 재현
-> 기술 도입
-> 수치 비교
-> 문서화
```

이 순서를 지켜야 “Kafka를 써봤다”가 아니라 “왜 Kafka가 필요했고, 도입 후 어떤 문제가 좋아졌는지 설명할 수 있다”가 된다.

## 한 줄 결론

Kafka와 Redis 외에도 반드시 학습해야 할 것은 `RPC`, `Indexer`, `PostgreSQL 정합성`, `Observability`, `Kubernetes`, `Security/Key Management`, `Rust Signer`, `Smart Contract`, `성능 테스트`다.

다만 지금 당장 모두 배우는 것이 아니라, Phase 2의 금융 백엔드 코어를 끝낸 뒤 실제 문제를 만나면서 하나씩 붙이는 방식이 가장 효과적이다.
