## Day24에서 구현하려는 코드

Day24 구현가이드는 아래 파일을 기준으로 한다.

```text
internal/settlement/reconciliation.go
internal/settlement/reconciliation_repository.go
internal/settlement/reconciliation_test.go
internal/settlement/reconciliation_repository_test.go
```

핵심 설계는 다음과 같다.

```text
Reconciliation은 자동 수정 기능이 아니다.
읽기 전용으로 Settlement와 Ledger를 비교하고 report를 만든다.

error:
검증 자체를 수행할 수 없는 실패
예: DB 연결 실패, context 취소, query 실패

ReconciliationIssue:
검증은 성공했지만 데이터 불일치를 발견한 결과
예: 금액 불일치, 통화 불일치, 수취인 불일치, ledger entry 누락
```

중요 비교 기준:

```text
settlement_items.ledger_entry_id -> ledger_entries.id
settlement item amount -> ledger entry amount
settlement item currency -> ledger entry currency
settlement item recipient_id -> ledger account owner_id
ledger account type == MERCHANT_PENDING
ledger entry direction == CREDIT
batch total_amount == settlement item amount sum
```

## 작업 규칙 요약

### 문서

```text
Git Markdown, Jira, Confluence는 Day 단위로 함께 생성/수정한다.
Git Markdown 상단에는 관련 Jira와 Confluence 링크를 남긴다.
Confluence 메인 페이지에는 Jira, 구현가이드, 실습산출물 링크를 남긴다.

다이어그램이 필요하면 다크 테마 PNG를 우선 사용한다.
```

### 구현가이드

```text
Day25 이후에는 Flow-first 방식을 우선한다.
실제 운영 시나리오와 대표 진입점 함수/service 흐름을 먼저 보여준다.
그 다음 해당 흐름 때문에 필요한 entity, table, repository, interface를 설명한다.

생성/수정하는 파일은 전체 코드를 제공한다.
전체 코드는 가능한 경우 details 접기 구조로 제공한다.
확인만 하는 코드는 필요한 메서드나 쿼리만 발췌해도 된다.
```

## 새 대화의 Codex에게 남기는 주의

사용자는 현재 Go, SQL, 블록체인 도메인을 함께 학습 중이다.

설명할 때는 아래 방식을 유지한다.

```text
1. 먼저 결론을 말한다.
2. Java 개발자 관점의 비교를 자주 제공한다.
3. Go의 포인터, receiver, interface, context, 테스트 helper는 반복해서 자세히 설명한다.
4. 영어 용어는 한글 의미를 같이 적는다.
5. “왜 이 기능이 필요한지”를 코드보다 먼저 놓치지 않는다.
6. 단순 문서 작업보다 실제 기능 흐름과 테스트를 우선한다.
7. Entity-first보다 Flow-first를 우선한다.
   service/processor의 대표 진입점과 전체 호출 흐름을 먼저 잡고,
   그 다음 entity/repository/interface를 필요한 이유와 함께 설명한다.
```

현재 중요한 학습 축:

```text
Ledger:
돈의 이동을 이중 entry로 기록하는 원장

Settlement:
확정된 ledger entry를 지급 가능한 묶음으로 계산하고 상태 전이하는 과정

Reconciliation:
Settlement 결과와 Ledger 원본이 일치하는지 다시 검증하는 읽기 전용 대사 과정

Indexer:
블록체인 이벤트를 읽어 내부 DB 상태로 안전하게 반영하는 프로세스
```

이 문서를 읽은 새 대화의 첫 응답은 장황하게 다시 설명하기보다, 현재 상태를 짧게 요약하고 바로 다음 작업으로 들어가면 된다.
